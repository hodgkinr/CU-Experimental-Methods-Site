Title: The Aerospace Modeling & Validation Framework
Talking Point: Welcome to ASEN 3501. Before we touch a single piece of hardware or write a single line of MATLAB, I want to establish the professional identity this course is building toward and the framework that organizes everything we will do. You are training to be a test engineer. Test engineers design tests, characterize systems, and validate models. The AMVF is the framework test engineers use, uncertainty is the language test engineers speak, and the ability to argue for a result numerically and with bounded confidence is the output test engineers produce. That identity stake starts today, and I will echo it at the opening of every lab segment this semester.

Your other coursework builds your capacity to generate predictions — to say "based on this model, the system should behave like this." That skill is genuinely valuable. slide 4: The AMVF — Two Branches, One Validation Goal
(Image: Clean AMVF flowchart on white background with gold highlights. Two parallel vertical branches. Left branch labeled "Mathematical Modeling Branch" — nodes top to bottom: System/Phenomenon of Interest → Conceptual Model → Mathematical Model → Computational Model → Simulation Outcomes. Right branch labeled "Physical Modeling Branch" — nodes top to bottom: System/Phenomenon of Interest → Physical Realization → Experimental Measurement → Data Reduction → Experimental Outcomes. Both branches share the top node and converge at the bottom on a node labeled "Quantitative Comparison." Below Quantitative Comparison: two labeled arrows — left arrow "Validation" pointing back up to Mathematical Model, right arrow "Experimentally Informed Model" with a feedback loop icon. Uncertainty quantification is shown as a vertical bar spanning both branches. Clean and high-contrast.)
Talking Point: This is the Aerospace Modeling and Validation Framework — the AMVF — and it is the organizing spine of this course. The left branch is what your other classes develop: you take a physical phenomenon, build a conceptual model, translate it into mathematics, run a simulation, and produce a prediction. The right branch is what this course teaches: you take the same phenomenon, design a physical realization of it, instrument and measure it carefully, reduce the data, propagate uncertainty, and produce an experimental outcome. Validation is what happens when both branches meet at the bottom of this diagram. The critical insight is that neither branch is complete without the other — a simulation without experimental grounding is an educated guess, and data without a model to compare against is just observation. The meeting point is where engineering credibility lives.

slide 5: Verification vs. Validation — Not the Same Thing
(Image: Two clearly separated panels side by side with a bold vertical divider. Left panel, labeled "Verification" with a blue checkmark icon: title text "Are we solving the equations correctly?" Below, a code snippet on a screen being compared to an analytical reference solution. Caption: "Internal consistency — does the code do what the math says?" Right panel, labeled "Validation" with a gold target-and-arrow icon: title text "Are we solving the right equations?" Below, a graph showing a model prediction curve overlaid with experimental data points and error bars. Caption: "External correspondence — does the model represent physical reality?" Below both panels: "You can have a perfectly verified simulation and a completely invalid model.")
Talking Point: Verification and validation are two distinct activities that get conflated constantly, and I want to be precise about this from Day 1. Verification asks whether we are solving the equations correctly — it is a question about numerical implementation, about whether the code faithfully represents the mathematics. Validation asks whether we are solving the right equations — whether the mathematical model actually corresponds to the physical phenomenon it claims to describe. You can write flawless, bug-free code that perfectly implements a model, and if that model has wrong physics in it, you have a perfectly verified, completely invalid simulation. This course lives almost entirely in the validation territory. The right branch of the AMVF is how we gather the physical evidence needed to answer the validation question. A concrete example of the distinction: if a structural analysis code correctly solves the finite element equations but uses the wrong boundary conditions for the actual physical setup, the code is verified (correct solution of the equations) but the model is not validated (does not represent the physical reality of the structure).

slide 6: Outcomes vs. Results — A Distinction You Will Use Every Week
(Image: A vertical flow diagram on a light background. Four steps shown top to bottom, connected by arrows. Step 1: "Raw Sensor Output" — a noisy voltage-vs-time trace. Step 2: "Data Reduction." Step 3: "Result" — "9.76 m/s²." Step 4: "Outcome" — "9.76 ± 0.14 m/s² (expanded uncertainty, k=2, method stated)." A branch points to "Quantitative Comparison." A note reads "Label the interval type; do not use CI as a generic synonym for uncertainty.")
Talking Point: The AMVF maps physical measurements to experimental outcomes — not experimental results. These are not the same thing. A result is a number that comes out of your data reduction: 9.76 meters per second squared. An outcome is a result that also carries its uncertainty, a stated confidence level, and a method statement — 9.76 plus or minus 0.14 meters per second squared at 95% confidence, computed by partial derivative propagation. The distinction matters because the outcome is what allows comparison. Without the uncertainty bounds, I cannot determine whether 9.76 agrees with a model that predicts 9.81. With the bounds, I can make a defensible quantitative argument. From this point forward, every numerical claim you make in this course must be expressed as an outcome.

slide 7: Uncertainty Is Mandatory — Here Is Why
(Image: A horizontal number line showing a model prediction at a fixed point — vertical dashed line labeled "Model Prediction: 42.3 N·m." Two experimental outcomes are shown as horizontal bars indicating uncertainty ranges. Outcome A: narrow bar centered at 44.1, labeled "44.1 ± 0.4 N·m — gap exceeds uncertainty, significant discrepancy." The bar does not overlap the model prediction line. Outcome B: wider bar also centered at 44.1, labeled "44.1 ± 2.8 N·m — agreement within uncertainty." This bar overlaps the model prediction line. Below both: "Same measured value — completely different engineering conclusions." A bold caption: "Uncertainty is not a footnote. It is the argument.")
Talking Point: Here is the most important visual in this lecture. Both outcomes show the same measured value — 44.1 Newton-meters. The model predicts 42.3. Do they agree? The answer depends on how the uncertainty and the comparison criterion are characterized. In Outcome A, the uncertainty is tiny — the gap between 44.1 and 42.3 is much larger than the bounds, and I have a significant discrepancy I need to explain. In Outcome B, the uncertainty is larger — the gap falls within the bounds, so agreement may be defensible for the stated purpose. Professional measurement-uncertainty guidance such as ASME PTC 19.1 and the GUM provides the basis for reporting these bounds. Uncertainty is not a sign of weakness. It is part of the argument.

(Image: The full AMVF flowchart from slide 4, now with a gold shaded overlay covering the Physical Modeling Branch nodes (Physical Realization, Experimental Measurement, Data Reduction, Experimental Outcomes) and the Quantitative Comparison node. Talking Point: For the next five weeks — Phase I of this course — your job is to walk the right branch of the AMVF with discipline. The mathematical model is given to you. Your job is to take the physical phenomenon, design and execute the measurement, reduce the data, propagate your uncertainty, and produce an experimental outcome that can be compared to the model prediction. You are not designing the experiment from scratch yet — that is Phase III. You are not developing the mathematical model independently — that is Phase II. But Phase I is not just a warmup. The physical modeling branch is where experimental credibility is earned or lost. A perfectly good mathematical model cannot be validated by a carelessly executed experiment. Phase I builds the foundation everything else rests on.

slide 9: The AMVF Connects to Professional Engineering Standards
(Image: The AMVF flowchart shown as a central hub with four satellite documents connected by arrows labeled "informed by." Satellite 1: ASME PTC 19.1 / GUM — measurement uncertainty. Satellite 2: ASME V&V 10-2006 — legacy computational solid mechanics guide. Satellite 3: ASME V&V 20 — computational fluid dynamics and heat transfer. Satellite 4: relevant AIAA or domain measurement standard. Below: "Choose and explain the standards relevant to the intended use." Clean professional layout.)
Talking Point: The AMVF is a course synthesis informed by professional practice; no single standard governs every experiment. For measurement uncertainty, our main basis is Coleman and Steele together with ASME PTC 19.1 and the GUM. ASME V&V 10-2006 is the legacy Guide for Verification and Validation in Computational Solid Mechanics; ASME V&V 20 addresses computational fluid dynamics and heat transfer. We use those V&V documents as domain examples of intended-use and model-credibility reasoning, not as universal experimental procedures. In E3, you will identify the standard or handbook that actually fits your model, measurement domain, and intended use, then annotate what it contributes and what it does not establish.

slide 10: The Experimentally Informed Model
(Image: A cyclical three-node loop diagram. Node 1 (top): "Mathematical Model" — an equation displayed, e.g., F = ma with some terms highlighted. Arrow down-right to Node 2 (right): "Experiment" — a test article with sensors, arrows labeled "predicts expected outcome." Arrow down-left to Node 3 (bottom): "Experimental Outcome" — a graph with data points and uncertainty bars alongside a model line, labeled "measured, with uncertainty." Arrow up-left back to Node 1, labeled "informs and updates model." The loop conveys iteration. Inset in the lower right corner: a small "before/after" graphic showing a model prediction curve (dashed) that initially misses the data, then an updated curve (solid) that passes through the data within uncertainty — labeled "Experimentally Informed Model.")
Talking Point: There is a common misconception I want to address on the first day: the goal of an experiment is not to confirm the model. The goal is to learn from the comparison. When your experimental outcome disagrees with the model prediction, that discrepancy is not a failure — it is information. It tells you what physics the model is missing, or what physical effect you did not account for in your experimental setup. The most valuable output of the right branch of the AMVF is not a verdict of "confirmed" or "rejected" — it is an experimentally informed model, one that has been updated based on what the data revealed. For example, if you measure a pendulum's period and find it increases at larger amplitudes, the data tells you the simple-pendulum model is missing an amplitude-dependent physics term — that is the experimentally informed model insight. This concept runs through all five weeks of E1 and becomes the central deliverable in E2 and E3. In the first lab session this week, you will begin designing your canned experiment with this outcome explicitly in mind.

slide 12: Evaluator Moment — Classify This Claim
(Image: A single quote box on a clean white background, styled to look like an excerpt from an engineering report. The text reads: "We ran the finite element simulation and compared the deflection predictions to our physical test results. The predictions agreed with the measurements to within 2.5%. The structural model is verified." Below the quote, a single prompt in bold: "Is this claim defensible? What, if anything, is wrong with it?")
Talking Point: Take two minutes and evaluate this claim. [Pause — allow student responses.] The key issue is vocabulary: agreement with physical test data is validation, not verification. Verification checks whether the code correctly solves the equations — it is an internal consistency test. Validation checks whether the model correctly represents physical reality — it requires comparison to physical evidence. This claim has done exactly the right experiment and reached exactly the wrong conclusion about what it demonstrates. A test engineer reading this report would flag the word "verified" immediately. If you can catch that error in two minutes on a report you have never seen before, you are doing test engineering. That is the skill this course is building.

slide 13: Closure
The Aerospace Modeling and Validation Framework is not a diagram you memorize and move on from. It is the decision-making structure that a test engineer returns to every time they need to argue that a result is credible — when they design an experiment, when they argue for or against agreement between data and model, and when they plan a V&V strategy for a complex system. Today you have seen where this course sits in that structure, why verification and validation are different things, and why uncertainty is not optional but mandatory. Over the next fifteen weeks you will internalize all of it — not by memorizing the framework but by using it, week after week, to make defensible engineering claims about physical systems. That is the skill employers pay for. That is what we are building together. Welcome to Aerospace Experimental Methods.
