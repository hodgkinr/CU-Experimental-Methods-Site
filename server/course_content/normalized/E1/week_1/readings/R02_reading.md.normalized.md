# GitHub Basics for Course Documentation
## Your Experiment Lives Longer Than Your Lab Session — Here's Where to Put It

An undocumented experiment is an experiment that never happened. In professional aerospace practice, an experiment that exists only in memory is worth nothing to the next engineer who needs to reproduce, extend, or audit your work.

---

Engineering documentation has two jobs: it preserves what you did and why you did it, and it makes your work checkable by someone who was not there. Professional test engineers maintain meticulous records not because they enjoy paperwork, but because the value of a measurement result depends entirely on the ability of a third party to evaluate it independently. A voltage reading without context is noise. A voltage reading with a documented procedure, sensor specification, calibration record, and uncertainty estimate is evidence.

**GitHub** is a platform for hosting and sharing documents using version control, a system that tracks every change you make to a file over time, along with who made the change and when. In software development, this is used to manage code across large teams. In this course, you'll use it more simply: as a public-facing portfolio where each experiment you conduct has a documented home and a traceable history.

The mechanics are straightforward. A **repository** (repo) is a folder with version history: a project folder that remembers every saved version. A **README.md** is a text file written in **Markdown**, a lightweight formatting language, that appears on the repository's front page and describes what the project is. You'll write your experiment documentation in Markdown: plain text with simple syntax for headers (`# Title`), bold (`**term**`), and lists (`- item`) that renders cleanly in the browser. No special software required beyond a text editor and a browser.

The workflow for adding files has three steps: *stage* (select which files to include in this snapshot), *commit* (save a labeled snapshot with a short note about what changed and why), and *push* (send that snapshot to GitHub's servers so it is accessible online). For this course you won't need the full depth of version control (branches, pull requests, merge conflicts), but understanding the commit as "a labeled save with a note about what changed" is the fundamental concept. Your commit messages are part of your documentation narrative; write them as though you're explaining your lab session to a colleague six months from now.

Getting started requires creating a free account at github.com, creating a new repository for ASEN 3501, and uploading your first file, which will be the pre-lab design work from this week's Tier 1 Phase 1 design session. That first file should include your experimental hypothesis, your identified variables, and a description of your annotated AMVF diagram. The repository structure you establish now will carry forward through all subsequent experiment deliverables. Use AI tools freely to help with syntax questions, Markdown formatting, or workflow steps. This is exactly the kind of mechanical task where AI augmentation is efficient and appropriate.

One habit worth building from the start: GitHub is a record of your process, not a drop box for final products. A repository with one massive commit at the end of the semester tells no story. Frequent, small commits with clear messages create a record that shows how your thinking developed. That process record is a professional artifact. Engineers and employers who look at your GitHub portfolio are reading your technical judgment over time, not just the last thing you submitted.
